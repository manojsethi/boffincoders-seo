# 01. Product Principles: SEO Action System

## Goal

Move the product from audit reporting to SEO execution.

The product should not stop at:

> This page has issues.

It should answer:

> What is the root cause, what should be fixed, who should fix it, what evidence supports the recommendation, and how will we validate the result?

## SEO Execution Model

Use this mental model throughout the product.

| Product Concept | SEO Product Equivalent |
|---|---|
| Signal | Issue, traffic drop, ranking drop, poor CTR, thin content |
| Root cause analysis | Explanation backed by crawl, audit, GSC, GA4, CWV, keyword, or page evidence |
| Recommendation | Recommended fix/action |
| Execution detail | Priority, scope, effort, owner |
| Validation | Re-crawl, re-audit, GSC/GA4/CWV monitoring |
| Outcome | Verified fix, improved clicks, better CTR, better conversion |

## Product Rule

Audit output is not enough.

A useful SEO agency tool must produce:

- recommendations.
- keyword priorities.
- content improvement plans.
- internal link recommendations.
- technical handoff tasks.
- validation methods.
- progress tracking.
- client-ready explanations.

## Recommendation Types

Start with these:

- technical fix.
- content improvement.
- title/meta rewrite.
- internal link action.
- schema action.
- keyword/page targeting action.
- page creation action.
- page consolidation action.
- conversion improvement.
- CWV/performance improvement.
- report/client communication action.

## Recommendation Quality Bar

Every recommendation must include:

- root cause summary.
- root cause.
- evidence.
- recommended action.
- expected impact.
- effort.
- owner.
- priority.
- validation method.
- linked pages.
- linked keywords when relevant.
- linked goals when relevant.

## Must Change Standard

Only use `must_change` when evidence is strong.

Strong evidence examples:

- important page is noindex.
- wrong canonical points away from target page.
- GSC shows high impressions and very low CTR.
- query ranks but wrong page is ranking.
- multiple pages compete for same keyword.
- commercial page has no clear CTA.
- content does not address mapped keyword intent.
- important page has poor CWV.
- service/product page is missing key schema where applicable.
- GA4 shows high organic traffic but weak conversion/engagement.

Weak evidence examples:

- AI feels the copy is generic.
- page could maybe be longer.
- title is not perfect.
- competitor might have more content but no competitor data exists.

Weak evidence should become `consider_improving`, not `must_change`.

## AI Usage Principle

Local AI:

- classify.
- summarize.
- explain.
- group.
- draft recommendations.

External premium AI:

- content brief expansion.
- final draft writing.
- high-stakes strategy explanation.
- client-ready polished content.

Rules/data:

- severity.
- validation.
- evidence.
- scoring.
- must-change threshold.

## Analyst Control

Every recommendation should support:

- approve.
- edit.
- reject.
- assign owner.
- set due date.
- mark planned.
- mark implemented.
- verify after re-audit.

## Completion Criteria

This module is successful when an analyst can open a project and answer:

1. What should we fix first?
2. Why does it matter?
3. Which keyword/page/goal does it support?
4. Who should do it?
5. What exact change is needed?
6. How do we verify it worked?
